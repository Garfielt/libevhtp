The reality of Libevent's http API: It was originally written as an example use-case for Libevent itself. Niels (the author of Libevnt) extended the example further, mainly to implement "secproxy" (I can't remember the name, but it was something like that, SORRY Niels!)

In short: Libevent's evhttp API is not NGINX-IN-A-BOX.

But users have those expectations.

Unfortunately it falls short in many areas: flexibility, readability, and most importantly: performance, and agility.

Libevhtp strives to create a "NGINX-IN-A-BOX", but allowing the developer to decide how deep down the rabbit hole they want to go. You can create a fully functioning multi-threaded HTTPS service with 4 lines of code.

But much like apache's plugin API, the developer has a considerable amount of control over WHAT happens and WHERE it happens. That is; A developer can optionally create hooks into every phase of the request processing cycle. For example:

A developer can pause the processing after the request headers have successfully been parsed, run a background job (hopefully using the deferred mechanisms defined within libevthr), and finally resuming the request processing once the job has completed.

But that's just one simple example of the many things you can do with Libevhtp. Be creative, and have fun.
